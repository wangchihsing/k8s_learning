💡 學習收穫

Deployment 配置
  • replicas: 副本數量
  • selector: 標籤選擇器
  • strategy: 更新策略
  • containers: 容器設定
健康檢查機制
  • livenessProbe: 存活探針
  • readinessProbe: 就緒探針
  • initialDelaySeconds: 延遲時間
  • periodSeconds: 檢查間隔
資源管理
  • requests: 請求資源
  • limits: 限制資源
  • memory: 記憶體設定
  • cpu: CPU 設定
高可用配置
  • podAntiAffinity: 反親和性
  • topologyKey: 拓撲鍵
  • weight: 權重設定
  • 滾動更新策略

🏢 房子比喻系統建立

+
Port = 房間號碼

🌳 端口知識樹成長

基本概念：一個 port 同時間只能給一個程式使用
常見端口：80(HTTP), 443(HTTPS), 22(SSH), 3306(MySQL)
Kubernetes 端口：containerPort, port, targetPort, nodePort
端口衝突：address already in use 錯誤的原因和解決

🤔 邏輯思考展現

「所以只要 ip 不一樣，一樣的 port 就沒問題了嗎」

✅ 完美理解！

你的理解完全正確！這展現了優秀的邏輯思維能力。

192.168.1.100:80 ✅ (A電腦的80端口) 192.168.1.200:80 ✅ (B電腦的80端口) 10.0.0.50:80 ✅ (C電腦的80端口) // 不同 IP = 不同大樓的相同房間號 = 沒問題！

🔧 第三階段：實戰故障排查

🎯 明確目標

❌ 遇到問題

直接訪問失敗：curl: (28) Failed to connect

🔍 系統化診斷

你學會了分層測試的重要性：

  • ✅ Kubernetes 內部測試
  • ✅ Minikube 內部測試
  • ✅ 虛擬機到 Minikube 測試
  • ❌ Mac 到虛擬機測試

💡 問題定位

發現問題出在虛擬機軟體的網路限制

🛠️ 解決方案嘗試

# 方案1: iptables sudo iptables -t nat ... ❌ 失敗
→
# 方案2: socat 31000 sudo socat TCP-LISTEN:31000 ... ❌ 部分成功
→
# 方案3: 換端口 sudo socat TCP-LISTEN:8090 ... ✅ 完全成功！

🎉 重大突破時刻

「所以最後那個命令是把虛擬機的8090端口跟nodeport的31000端口連再一起才成功的嗎」
